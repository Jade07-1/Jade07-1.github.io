为什么需要 Ray#

在 AI 和大数据任务日益增长的今天，快速构建高性能、易维护、可扩展的分布式系统成为每个工程团队的核心需求。但传统方案往往存在以下痛点：

  • 需要两套代码（本地开发 + 上线集群）
  • 依赖消息队列或手动调度
  • 状态管理复杂，需要自己加锁或用数据库
  • 资源调度不灵活，难以混部 CPU/GPU 任务
  • 部署运维复杂，需手动打包、上传、配置 Ray 的出现，正是为了解决这些问题 。它的核心魔法在于：让分布式编程像写 python 本地程序一样简单。只需要一行装饰器@ray.remote，就能让函数从“单机运行”变成“集群并行”。

一句话定义 Ray#

Ray 是一个通用的分布式计算框架，支持任务并行、Actor 模型、资源调度、共享内存及容错等核心能力，让开发者便捷的构建复杂的分布式应用。

Ray 的核心能力#

  • 任务并行 ：自动将函数调用分布到多个节点上执行，以高效利用资源。
  • Actor 模型 ：支持构建有状态服务，如缓存和模型服务，简化状态管理。
  • 资源调度 ：灵活指定 CPU、GPU、内存等资源需求，实现精细化控制。
  • 容错机制 ：自动重试失败的任务，维护系统稳定性。
  • Python-first API ：友好的 Python 接口，同时支持 Java 和 C++。
  • 高性能数据传输：支持共享内存和跨节点数据传输，提升数据处理效率。
  • 云原生集成 ：兼容 Kubernetes、Docker、云存储。 Ray 的设计理念： 让开发者专注逻辑和资源需求，而将调度、容错、传输、内存管理等决定性因素交由 Ray 自动处理。

Ray 的定位#

对比对象 Ray 的优势 Celery

  • Ray 作为分布式计算引擎，不依赖消息队列，内置调度机制，灵活构建复杂的服务；
  • Celery 是任务队列，需要依赖消息队列（如 Redis、RabbitMQ） Spark
  • Ray 延迟更低、支持有状态、适合 AI 场景、实时计算；
  • Spark 是大数据批处理引擎，更适合批处理、ETL等离线大作业 Kubernetes Job
  • Ray 是应用内的调度器，支持细粒度资源控制（如 0.5 个 CPU）、Actor 服务、多任务协同；
  • K8s 是基础设施调度，无法精细控制任务级资源，无法支持 Actor，适合短期的批处理任务 因此，Ray 不是 Spark、Kubernetes Job 的替代品，而是填补它们之间的空白 。 它更像是一个应用级的调度引擎 ，能解决 AI 工作流中遇到的“最后一公里”问题 —— 比如 ETL 后的数据如何高效调度到训练节点，如何在 GPU 上并行处理多模态数据，如何构建低延迟、高并发的推理服务等。

Ray 的工作流程简述#

  1. 定义一个函数或类，加上 @ray.remote 装饰器。
  2. 调用 .remote() 方法提交任务到集群。
  3. GCS（全局控制服务）调度任务到合适的节点。
  4. Raylet（本地调度器）在节点上调度任务或 Actor。
  5. 执行完后，结果存储在 Object Store 中，返回 ObjectRef。
  6. 其他任务或 Actor 可以通过 ray.get() 或 ray.wait() 获取结果。 Ray 的核心架构：一个头节点负责调度，多个工作节点负责执行 Task 和 Actor。
  • GCS：负责全局调度、元数据管理、Actor 生命周期管理
  • Raylet ：节点本地调度器，负责任务分发、资源管理
  • Object Store ：分布式共享内存，用于 Task/Actor 之间高效数据传输 [图片]

Ray 的核心优势#

Ray 的诞生背景：为了解决 AI 和大规模数据处理场景下，传统框架（如 Spark、Celery）在任务调度、状态维护、资源利用等方面存在多个痛点。 Ray 的设计初衷：为新兴 AI 应用（如深度学习、强化学习和超参数调优）提供一个高性能、易用且灵活的底层框架，从而提升开发和计算效率。

  1. 异构资源调度：灵活支持 CPU/GPU 混部
  • 问题：复杂 AI Pipeline 不同阶段资源需求差异大（如预处理用 CPU、训练用 GPU），传统框架调度不灵活。
  • Ray ：允许用户在 Task 和 Actor 上直接指定资源需求（如 @ray.remote(num_gpus=1)）。调度器能精确分配任务到相应资源的节点上，实现高效混部。
  1. 细粒度任务调度：微秒级响应，支持高频计算
  • 问题：如 Spark 等框架调度延迟高，难以应对强化学习、超参搜索等场景中数百万微小任务的调度需求。
  • Ray：支持微秒级的任务调度，每秒可调度数百万个任务，资源利用率高，适合细粒度、高并发的 AI 场景。
  1. 原生支持有状态计算：Actor 模型
  • 问题：Spark 等框架侧重无状态任务，状态维护需依赖外部系统（如 Redis），开发复杂、性能受限。
  • Ray：引入了 Actor 模型，允许开发者像编写普通 Python 类一样创建有状态服务。Ray 负责管理 Actor 的生命期，使得实现参数服务器和模拟环境变得简单高效。
  1. 统一编程模型：覆盖 AI 全流程，一套代码跑 everywhere
  • 问题：构建完整 AI 系统通常需要多个工具，增加了架构复杂性及维护成本。且需编写两套代码，一套用于本地，另一套用于服务器，重复开发增加调试压力。
  • Ray：提供统一编程模型，通过 Task 和 Actor 实现数据处理、模型训练和部署，简化架构：
    1. 数据处理：通过 Ray Core 和 Ray Data 处理多模态数据和分布式 ETL。
    2. 训练与调参：Ray Train 和 Ray Tune 整合 PyTorch/TensorFlow，实现容错训练和超参数搜索。
    3. 模型部署：Ray Serve 实现在线服务。 一套代码哪里都能跑，只需要加一个 @ray.remote ，本地调试 → 集群运行，无需改代码。
  1. 简化任务提交与依赖管理：实现更轻量的开发
  • 问题：传统框架提交任务需手动打包环境、上传依赖，流程复杂。
  • Ray：支持运行时安装依赖（pip、conda）、支持自定义环境模板（如预装 CUDA、特定库）以及自动并行处理大数据，无需手动拆分（Ray Data）。
  1. 云原生友好：弹性伸缩，统一部署体验
  • 问题：云上部署需大量适配代码，资源利用率低。
  • Ray：支持自动扩缩容，任务高峰自动扩容，空闲时释放资源，节省成本。且支持运行在物理机、K8s、公有云等多种基础设施上。云存储可挂载为本地路径，代码无需修改即可读写（如 AWS S3、OSS）。
  1. 高度灵活：构建你想要的分布式系统
  • 问题：Spark、Flink 等框架抽象固定，难以支持高度定制化的 AI 系统。
  • Ray：提供的是一个通用的分布式计算原语 （Task、Actor、Object Store），可以像“搭积木”一样自由组合，将不同的分布式组件、库和逻辑粘合在一起，构建任意复杂的分布式逻辑。

核心编程模型：Task 与 Actor#

这是 Ray 最核心的两个概念

Task#

Task 是 Ray 中最小的执行单元，是一个无状态的远程函数调用。 它可以在集群中的任意节点上执行，适合做并行计算、批处理、流水线任务。示例代码：

加上 @ray.remote，这个函数就变成了可以在集群任意节点运行的 Task#

@ray.remote def square(x): time.sleep(1) return x * x

.remote() 是异步调用的，它会立即返回 ObjectRef#

此时主程序不阻塞，4 个任务在后台并行跑#

futures = [square.remote(i) for i in range(4)]

ray.get() 拿着 ObjectRef 去取结果，如果结果没算好，这里会等待#

results = ray.get(futures) 如果我们的任务需要共享状态，例如维护一个全局计数器或加载模型实例，可以使用 Ray 的 Actor 模型。

Actor#

Actor 是 Ray 中的有状态抽象，是一个远程类实例。 它在集群中运行一个独立进程，可以维护状态，并通过 .remote() 方法调用实现远程方法执行。适用于 模型推理服务（加载模型太慢，不能用完即毁）、全局计数器、训练参数服务器。示例代码： @ray.remote class Counter: def init(self): self.count = 0 # 这里的状态会被保留

def increment(self):
    self.count += 1
    return self.count

创建一个 Actor 实例，它在集群某台机器上启动了一个进程#

counter = Counter.remote()

调用方法，状态会累加#

f1 = counter.increment.remote() # count = 1 f2 = counter.increment.remote() # count = 2

print(ray.get([f1, f2])) Actor 保证了对状态的访问是串行的，不需要手动加锁就能保证线程安全。每个 Actor 都是一个独立的服务，可以在集群的任何地方被调用。 在实际业务中，我们的任务往往对资源有明确的需求，比如：图像处理任务需要 CPU、模型推理任务需要 GPU、某些任务可能需要特定的内存、加速卡、自定义资源（如 TPU、FPGA）等。 Ray 提供了非常灵活的资源调度机制，可以在定义 Task 或 Actor 时，通过参数直接指定所需资源，Ray 会自动调度到具备相应资源的节点上运行。如@ray.remote(num_cpus=2, num_gpus=1)

Ray 的生态与架构#

Ray 的架构分为基础设施层 、核心执行层（Ray Core） 和 AI 工作流层（Ray AIR）这三层 ，每层职责分明，协同调度任务。它连接了底层资源与上层 AI 逻辑，让开发者专注于“写什么”，而不是“怎么运行”。 [图片]

基础设施层#

  • Ray 可部署在物理机、云主机、Docker、Kubernetes 等基础设施之上
  • Kubernetes 提供节点级调度，Ray 负责应用级调度
  • Ray 自动管理节点资源，开发者无需关心底层是哪台机器

Ray Core#

  • 包含 Ray 的核心组件：GCS、Raylet、Worker、Object Store
  • 提供 Task、Actor、Object Store、Placement Group 等核心 API
  • 是所有上层 AI 应用的基础

Ray AIR#

  • 基于 Ray Core 层的机器学习库，这层是做机器学习 Pipeline
  • 面向AI开发者，提供了一整套面向 AI 工作流的工具链 ，统称为 Ray AIR（AI Runtime）： 模块 功能 Ray Data 分布式数据处理，支持多模态数据（文本、图像、视频等） Ray Tune 分布式超参数调优（网格搜索、贝叶斯优化） Ray Train 分布式模型训练，集成 PyTorch、TensorFlow等框架 Ray Serve 模型服务部署，支持高并发、低延迟、版本控制、A/B测试 Ray RLlib 强化学习库，内置主流算法（ PPO、DQN、IMPALA 等） 这些模块共同构成了一个完整的 AI 工作流平台，从数据处理到训练、调优、部署、服务 ，都可在 Ray 上完成。

如何快速上手 Ray#

以下是本地单机测试使用 Ray 步骤

安装 Ray#

Step 1: 安装 Ray#

pip install ray 感兴趣的话，可以自行本地编译。 Ray 官方编译指导手册 深入理解其混合架构、掌握模块依赖与构建流程。

编写并运行第一个 Ray 程序#

import ray import time

Step 2: 初始化并写代码#

ray.init()

@ray.remote def square(x): time.sleep(1) return x * x

并行执行 Task#

futures = [square.remote(i) for i in range(4)] results = ray.get(futures)

执行结果：[0, 1, 4, 9]#

print(“Task Results:”, results)

ray.shutdown()

查看 Ray Dashboard#

通过 Dashboard，可以实时看到任务的执行情况、资源使用率、日志等。

Step 3: 查看 Dashboard#

ray dashboard [图片]

可观测性与监控#

  • Dashboard：Ray 内置的可视化平台
    • 可查看：集群节点状态、任务执行进度、Actor 运行情况、资源使用统计。
  • 日志聚合：Ray 支持统一的日志收集
    • 通过 ray logs 命令即可查看远程任务或 Actor 的运行日志。
  • 指标与监控：通过 Prometheus + Grafana 可集成 Ray 的运行时指标
    • 如：CPU/GPU 使用率、任务延迟、内存占用、数据传输带宽。
  • Ray State API：提供命令行和 API 接口
    • 查看：当前运行的作业、Actor 状态、资源池使用情况、Object Store 数据统计。

小米 / 业界应用案例#

Ray 已在多个大厂应用，如小米、阿里、字节等，用于大模型训练、推理、多模态数据处理及高并发任务调度等。 Ray 的典型使用场景

  • 分布式模型训练与调参

  • 实时推理服务部署

  • 大数据 ETL 流水线

  • 分布式缓存、连接池、队列服务

  • 高并发任务调度（如爬虫、任务队列） case 1 - 在线机器学习中的应用 现在我们通过一个贴近实际业务的案例，来帮助理解 Ray 如何在真实生产环境中提升 AI 工作流的效率与性能。

  • 背景：一个实时推荐或广告投放场景 ，系统需要根据用户的实时行为（如点击、浏览等）快速做出决策，推送最合适的优惠内容。这种场景对模型的实时性、响应速度和训练效率提出了很高的要求。

  • 支撑这个业务场景的技术架构，整个流程以 Ray 为核心调度引擎 ：

    • 从数据流开始，通过 Ray 实现任务调度与资源管理。
    • Ray 创建并调度多个分布式任务（Tasks）和 Actor，其中一部分 Actor 被用于运行 TF Worker ，直接与共享内存交互，实现高效的特征处理与模型训练。
    • 同时，Ray 还管理了 参数服务器（PS） ，用于同步和更新模型参数，并将训练完成的模型部署到 Serving 节点 ，为前端应用提供实时推理服务。 [图片] 通过案例看出，Ray 不仅是一个底层的分布式调度引擎，更是连接数据、训练、服务与调度的统一平台 ，非常适合需要实时性、高并发和灵活调度 的 AI 场景。可以把复杂的机器学习流程变得更轻、更快、更可控，是 AI 工程化道路上不可多得的利器。 整个架构设计体现了 Ray 的几个关键能力：

  • 任务调度与资源管理 - Ray 能够将数据处理、模型训练、参数同步和模型服务部署统一调度，避免了多个系统割裂的问题；

  • 共享内存优化 - TF Worker 与共享内存 Object Store 共存，减少了数据复制和传输开销；

  • 端到端低延迟 - 从数据输入、模型训练到上线部署，整个流程仅需 3~5 分钟，极大提升了模型迭代效率；

  • 业务指标提升 - 与传统的离线模型相比，CTR 提升了 5%，相较于 Blink 也提升了 1%。 case 2- 大模型数据处理流水线 目前内部 Ray 在大模型数据处理的使用背景： 在小米的大模型训练体系中，数据处理链路是构建高质量训练样本的核心环节，涵盖了从原始数据抽取、清洗过滤、去重打标到最终分桶输出的全过程。由于链路中涉及大量非结构化数据处理、GPU 推理任务调度以及复杂业务规则执行，对系统的灵活性、稳定性和计算效率提出了较高要求。 Ray 凭借其轻量级的分布式抽象、统一的调度机制和良好的扩展性，成为我们数据处理流程中的关键基础设施。与传统以 Spark 为主的批处理架构相比，Ray 更擅长应对那些计算密集、任务粒度细、依赖 GPU 资源 的场景，例如模型打标和文件粒度的并行处理。它允许使用 Python 编程范式进行开发，同时无缝扩展到大规模集群，极大降低了开发与调试成本。 数据抽取（Ray）

    • Python 原生开发，迁移成本低
    • 支持文件粒度处理，便于应对长尾数据

  • 场景：

    • 开发调试便捷，支持从单机到集群的平滑过渡 核心价值：Ray 提升了非结构化数据处理的效率，灵活调度 GPU 推理任务，降低开发和调试成本，并增强数据处理链路的可控性。它成为构建统一 AI 工程链路的重要组成部分。 Ray 在小米大模型数据处理中的应用，不仅帮助我们打通了从数据准备、模型处理到训练服务的端到端流程，还构建了一个更加统一、灵活、可控的分布式计算平台。我们希望进一步推动 Ray 在大模型训练、推理和在线服务中的深度整合，打造一个以 Ray 为底座的 AI 工程化体系，为业务迭代与技术创新提供更强有力的支持。
